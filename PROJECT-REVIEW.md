## 1. Product context

- **What it does:** Emoji-first shopping list app with AI-assisted product management
- **For whom:** Single user / family — personal grocery shopping with emoji browsing and AI text import
- **Platform:** Mobile-first PWA (Next.js, responsive, service worker, manifest), deployed as web app
- **Stage:** Active development (v0.1.0, Feb 2026), feature-complete MVP with post-MVP planning underway
- **Tech stack:** Next.js 16 / React 19 / TypeScript / Tailwind CSS 4 / Prisma 7 / PostgreSQL / Zustand / TanStack Query / NextAuth v4 / Google GenAI + OpenAI / UploadThing / Vitest / Playwright
- **Deployment:** Vercel (serverless), PostgreSQL external, Upstash Redis for rate limiting

### Key user scenarios

1. **Browse & add products** — user opens categories, taps emoji products to add to their shopping list
2. **Search & quick-add** — user types product name, autocomplete suggests, item added instantly
3. **Shopping mode** — user walks through store, toggles items as purchased, adds notes
4. **Import from text** — user pastes a text list (e.g., from a message), AI parses and maps to products
5. **Manage lists** — user creates multiple lists, switches active list, views purchase history and restores past lists

### Restructuring details

**Offline strategy:** No client-side data layer beyond Zustand. Offline queue needs: (1) persistent client store (IndexedDB), (2) sync service with write queue and replay, (3) conflict resolution per entity. Minimal change: design `src/services/sync/` before attempting.

**Real-time collaboration:** Entirely request-response today. Real-time needs persistent connections (WebSocket/SSE), presence tracking, live update propagation. Recommend `architecture-first` for this subsystem.

**List sharing:** Data model assumes single-user ownership (`userId` on List). Auth checks (`session.user.id === list.userId`) are spread across route handlers with no abstraction — hard to extend to multi-user access.

## 6. Recommendations

### Architecture

#### R-1: Extract service layer for business logic

- **Priority:** high
- **Effort:** large
- **Addresses:** C-1 through C-7, C-9 through C-12
- **Description:** Create `src/services/` with domain-specific modules (`list-service.ts`, `product-service.ts`, `history-service.ts`). Move Prisma queries and business logic out of route handlers. Handlers should only: parse request → call service → format response. This is the highest-impact structural change — it unblocks testability, reusability (for future offline sync, sharing), and prevents further logic sprawl.

#### R-2: Split `src/lib/` into domain directories

- **Priority:** medium
- **Effort:** medium
- **Addresses:** C-8
- **Description:** `src/lib/` holds auth (5 files), AI prompts, validation, email, Prisma client, utils. Split into `src/lib/auth/`, `src/lib/ai/`, `src/lib/email/`, `src/lib/db/`. Keep `utils.ts` as-is (it's clean).

#### R-3: Define AI subsystem boundary before hardening

- **Priority:** medium
- **Effort:** small
- **Addresses:** D-4, future readiness (AI hardening)
- **Description:** Before adding rate limiting and caching, define what the AI subsystem owns and doesn't do. This prevents hardening work from further embedding infrastructure into handlers.

- **Priority:** high

- **Description:** All three major post-MVP features (sharing, collaboration, offline) require a data access layer that doesn't exist. Before starting any of them, define a repository or service pattern. Otherwise each feature will add more direct Prisma calls to route handlers.

### Development strategy

**Crossroads:** The project has completed MVP features and is deciding what comes next. The current structure (everything in route handlers, no service layer) worked fine for building the MVP quickly. But the roadmap items (AI hardening, offline, sharing, collaboration) all require structural foundations that don't exist yet.

**Options:**

1. **Continue feature-first:** Keep building features on the current structure. Each new feature adds more logic to route handlers. Fast in the short term, increasingly painful as features interact.

2. **Extract service layer first, then features:** Spend time extracting business logic into services before adding new features. Slows down the next 2-3 weeks but makes everything after that faster and safer.

3. **Hybrid — extract as you go:** When building the next feature (AI hardening), extract only the affected routes into services. Gradual improvement, but risks inconsistency.

**Recommendation:** Option 2 — extract service layer first. The project is at the classic post-MVP inflection point. The route-handler-as-everything pattern has reached its natural limit (131-line handlers with business logic, transactions, and raw SQL). Every planned feature will be harder to build and harder to test without a service layer. The investment is one-time; the payoff compounds with every subsequent feature.
